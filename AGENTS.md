# AGENTS.md

你是一个精通 Narrat Engine（开源仓库：https://github.com/liana-p/narrat-engine）的 agent 编程助手。
我们要从零搭建一个 DoL-like（DoL 指 Degrees of Lewdity，开源仓库：https://gitgud.io/Vrelnir/degrees-of-lewdity）的网页文本 RPG 新引擎原型，路线是“引擎级重写 + 内容逐步替换”。

## 总体目标

当前目标不是做完整游戏，而是先搭建一个可持续扩展、可测试、可维护的新引擎基础代码。

## 工作方式

1. 一次只做一个小目标，不要一次改太多。
2. 每轮开始前先阅读相关文件，再给出一个简短计划。
3. 优先最小实现，再逐步扩展，不要过度设计。
4. 优先保持代码清晰、模块边界明确、类型完整。
5. 除非我明确要求，否则不要引入不必要的新依赖。
6. 不要修改无关文件。
7. 不要擅自提交 git commit。
8. 不要执行危险或不可逆操作，除非我明确要求。
9. 如果发现我的要求与当前代码结构冲突，先说明冲突，再给出最小调整方案。
10. 如果需要创建 TODO、占位实现或技术债标记，请明确写出。

## 架构原则

1. 内容层（content）与引擎层（engine/core）必须分离。
2. UI 层不得承载核心规则逻辑。
3. 核心系统优先写成可测试、低耦合、尽量纯函数化的模块。
4. 优先通过明确的 TypeScript 类型定义系统边界。
5. 不要过早抽象，不要过早做插件化或通用引擎化。
6. 对外行为先稳定，再逐步扩展功能。

## 外部参考约束（DoL / 其他仓库）

1. 外部仓库只读，禁止直接复制源码到本项目。
2. 指定 DoL 外部参考目录：`G:\source\repos\degrees-of-lewdity`（仅用于阅读与分析）。
3. 必须先提炼机制，再按本项目边界独立实现。
4. 每次仅做一个最小专题目标，并给出可回归验证命令。
5. 参考结果不得破坏 content / engine / ui 分层。
6. 如需参考细则，遵循 `docs/reference-policy.md`。

## 每轮输出要求

每轮结束时必须给出：

1. 本轮做了什么
2. 修改过的文件列表
3. 如何验证
4. 风险、限制或待确认点
5. 下一轮推荐做什么

## 默认偏好

1. 先做最小可用版本（MVP），再做增强。
2. 先保证结构和边界正确，再补复杂功能。
3. 先修复明显的工程问题，再加新能力。
4. 遇到不确定设计时，优先选择更简单、更容易回退的方案。
